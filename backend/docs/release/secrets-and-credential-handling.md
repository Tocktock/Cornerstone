# Secrets and Credential Handling

## Rules

```text
1. Never commit .env files.
2. Never paste Notion tokens into chat, tickets, screenshots, or docs.
3. Never include tokens in release reports.
4. Never log raw connector access tokens, refresh tokens, client secrets, or OAuth codes.
5. Use environment variables or a secret manager for runtime secrets.
6. Use production-safe CONNECTOR_ENCRYPTION_SECRET before enabling production mode.
```

## Sensitive values

```text
NOTION_E2E_ACCESS_TOKEN
NOTION_CLIENT_SECRET
CONNECTOR_ENCRYPTION_SECRET
DATABASE_URL when it contains credentials
CONNECTOR_OAUTH_CALLBACK_URL if it contains tenant-specific details
```

## Live proof evidence

Live proof reports may include:

```text
status
counts
IDs generated by Cornerstone
trust labels
metric summaries
sanitized source title/text snippets when safe
```

Live proof reports must not include:

```text
Notion access token
OAuth authorization code
OAuth client secret
.env file contents
raw provider payload with private workspace content
```

## Production hardening note

The backend currently has an application-level connector encryption boundary. A production deployment should decide whether to use a secret-manager/KMS-backed encryption provider before broader rollout.

This limitation is accepted for backend MVP only if the deployment remains a controlled pilot with appropriate operational safeguards.
