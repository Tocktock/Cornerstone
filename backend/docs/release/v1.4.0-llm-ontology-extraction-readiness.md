# v1.4.0 LLM Ontology Extraction Readiness

## Release identity

```text
Version: v1.4.0
Theme: LLM ontology extraction
Primary doc: docs/41-llm-ontology-extraction-v1.4.0.md
```

## Goal

Ship the first runtime path that turns EvidenceFragments into candidate ontology objects:

```text
EvidenceFragments → ConceptCandidates / RelationCandidates
```

The release proves that Cornerstone can begin constructing an ontology graph from source-backed evidence while keeping all generated output candidate-only.

## Explicit non-goals

```text
- no candidate approval workflow
- no candidate rejection workflow
- no candidate merge workflow
- no automatic official Concept creation
- no automatic official ConceptRelation creation
- no live external LLM credentials
- no graph visualization UI
- no depth greater than 1
- no vector search
- no PDF/Office/binary extraction
```

## Trust boundary confirmation

`v1.4.0` does not alter the official ontology trust boundary:

```text
Extractor output is candidate-only.
ConceptCandidates are not official Concepts.
RelationCandidates are not official ConceptRelations.
No candidate is included in the official graph.
Human review is still required before official SSOT promotion.
```

## Operator-facing behavior

Example pilot loop:

```bash
curl -X POST "http://localhost:8000/v1/ontology/extraction-runs" \
  -H 'Content-Type: application/json' \
  -d '{
    "artifactIds": ["ARTIFACT_ID"],
    "focusConcept": "settlement",
    "requestedBy": "reviewer@example.com"
  }'
```

Expected behavior:

```text
- run status becomes completed
- ConceptCandidates are created with status=pending
- RelationCandidates are created with status=pending
- official ontology graph remains unchanged
```

## Risks

```text
- The bundled provider is rule-based and should not be presented as live LLM quality.
- Candidate quality depends on explicit evidence wording.
- Duplicate detection is limited to normalized names and aliases.
- Review and promotion are deferred to v1.5.0.
- External LLM prompt injection defenses must be revisited when live providers are added.
```

## Release decision

The release is ready if compile, targeted integration tests, release-candidate checks, and Alembic offline SQL generation pass, and if the package includes the v1.4.0 implementation document, release notes, readiness document, migration, service, API endpoints, and tests.
