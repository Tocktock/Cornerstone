# v2.1.0 Roadmap — Live LLM Ontology Provider

## Version goal

Add a configurable live LLM ontology extraction provider that reads reviewed or reviewable `EvidenceFragments` and produces structured `ConceptCandidates` and `RelationCandidates`.

The provider must improve extraction quality beyond the local rule-based extractor while preserving the existing trust boundary:

```text
LLM output is candidate-only.
LLM output must never directly create official Concepts or official ConceptRelations.
Official graph mutation still requires evidence review and candidate review.
```

## User value

A user can upload or sync domain knowledge, such as settlement documentation, and Cornerstone can propose richer Concepts and depth-1 Relations than the local rule-based extractor. Reviewers still decide what becomes official.

## Confirmed non-goal

`v2.1.0` must not implement:

```text
- automatic Concept approval
- automatic Relation approval
- official graph mutation from LLM output
- frontend UI
- graph visualization contract
- graph depth above 1
- connector expansion
- batch review UX
- semantic/vector retrieval as source of truth
```

## Domain boundary

| Area | Boundary |
|---|---|
| Input | Existing `EvidenceFragments`, optional focus concept, extraction scope, and provider config. |
| Provider | External LLM call behind a provider interface. |
| Output | Pending `ConceptCandidates` and `RelationCandidates` only. |
| Validation | Backend validates structured output before persistence. |
| Official truth | Unchanged; official graph changes only through review/officialization gates. |

## Required behavior

The live provider must:

```text
1. Use structured JSON output.
2. Include evidence ids for every proposed concept.
3. Include evidence ids for every proposed relation.
4. Reject invalid relation types.
5. Reject missing or unknown evidence ids.
6. Reject self-relations.
7. Record provider name, model name, prompt version, and request metadata.
8. Support mock/fixture tests without live network calls.
9. Support gated live proof only when explicit environment variables are present.
10. Keep `local_rule_based` provider available as a deterministic fallback.
```

## Measurable acceptance checklist

| Check ID | Measurable acceptance condition | Evidence source |
|---|---|---|
| V210-01 | A live LLM provider interface exists and is separate from extraction orchestration. | Source module and tests. |
| V210-02 | Provider selection is configurable without code changes. | Config docs and tests. |
| V210-03 | Extraction run records provider name, model name, and prompt version. | API response and persistence tests. |
| V210-04 | Provider output is parsed through a strict schema. | Unit tests with valid/invalid payloads. |
| V210-05 | Concepts without evidence ids are rejected. | Negative tests. |
| V210-06 | Relations without evidence ids are rejected. | Negative tests. |
| V210-07 | Invalid relation types are rejected. | Negative tests. |
| V210-08 | Source-equals-target relations are rejected. | Negative tests. |
| V210-09 | Unknown evidence ids are rejected. | Negative tests. |
| V210-10 | LLM output creates only pending candidates. | Integration tests. |
| V210-11 | Official graph is unchanged after extraction. | Graph test before/after extraction. |
| V210-12 | Fixture/mock provider tests run without credentials. | CI test result. |
| V210-13 | Live provider proof is gated by explicit env vars. | Live proof doc and guard tests. |
| V210-14 | Local rule-based provider remains available. | Regression tests. |
| V210-15 | Release docs explain the candidate-only boundary. | Version doc and readiness doc. |

## Exit criteria

`v2.1.0` is complete only when a live provider can produce validated pending ontology candidates, and the official graph remains unchanged until human review.
