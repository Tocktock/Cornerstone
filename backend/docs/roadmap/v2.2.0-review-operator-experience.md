# v2.2.0 Roadmap — Review Operator Experience

## Version goal

Improve the API/CLI operator experience for reviewing ontology candidates before building a frontend UI.

The goal is to help reviewers answer:

```text
What candidates are pending?
Which focus concept do they affect?
Which candidates are high-confidence or low-confidence?
Which candidates are blocked by unreviewed evidence?
What would happen if I approve or merge this candidate?
```

## User value

Reviewers can process live LLM-generated ontology candidates more safely and quickly, while still preserving the official graph trust boundary.

## Confirmed non-goal

`v2.2.0` must not implement:

```text
- frontend UI
- visual graph layout
- automatic candidate approval
- automatic evidence review
- live LLM provider changes
- connector expansion
- graph depth above 1
- official graph mutation without explicit review action
```

## Domain boundary

| Area | Boundary |
|---|---|
| Candidate listing | Group and summarize existing pending candidates. |
| Review preview | Explain the expected effect of approve/merge/reject before mutation. |
| Review action | Use existing review services and officialization gates. |
| Official graph | Mutated only by explicit approved review actions. |
| UI | CLI/API only; frontend remains future scope. |

## Required behavior

`v2.2.0` should add review-oriented visibility:

```text
- review queue grouped by focus concept
- candidate counts by source/run/status
- evidence readiness summary
- approval preview
- merge preview
- blocked-candidate reasons
- CLI review summary command
- machine-readable next actions
```

## Exit criteria

`v2.2.0` is complete when reviewers can inspect, group, preview, and safely act on candidates through API/CLI without needing a frontend.
