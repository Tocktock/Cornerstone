# Cornerstone v1.5.0 — Ontology Candidate Review Workflow

## Purpose

`v1.5.0` implements the first human review workflow for ontology candidates.

Cornerstone can now take the `ConceptCandidate` and `RelationCandidate` objects generated by `v1.4.0`, let an authorized reviewer edit, reject, approve, or merge them, and only then promote approved objects into the official ontology graph.

This version is the trust-boundary bridge between:

```text
LLM/extractor proposal
→ human review
→ official Single Source of Truth graph
```

## Product goal

The goal is to make candidate review explicit, auditable, and safe:

```text
EvidenceFragments
→ ConceptCandidates / RelationCandidates
→ reviewer edit/reject/approve/merge
→ official Concepts / official ConceptRelations
→ explainable depth-1 ontology graph
```

The reference user story remains the settlement example:

```text
Manual uploaded settlement notes
→ extraction proposes Settlement, Clearing, Reconciliation
→ reviewer approves Concepts
→ reviewer approves Clearing --precedes--> Settlement
→ official graph shows Settlement connected to Clearing with citations
```

## What this version is

`v1.5.0` is a backend review workflow release.

It adds:

```text
- edit ConceptCandidate
- approve ConceptCandidate
- reject ConceptCandidate
- merge ConceptCandidate into an existing Concept
- edit RelationCandidate
- approve RelationCandidate
- reject RelationCandidate
- merge RelationCandidate into an existing ConceptRelation
- audit events for review actions
- candidate review metadata
- promotion ids / merge target ids
```

## What this version is not

`v1.5.0` intentionally does not add:

```text
- live external LLM calls
- better extraction quality
- graph visualization UI
- batch review operations
- semantic/vector deduplication
- graph depth above 1
- automatic unreviewed promotion
- automatic evidence review
- automatic source-trust override
```

## Trust boundary

The primary rule remains unchanged:

```text
LLM/extractor output is candidate-only.
Official ontology graph objects require authorized human review and reviewed eligible evidence.
```

`v1.5.0` enforces this with these rules:

```text
1. Only authorized reviewers can edit, approve, reject, or merge candidates.
2. Only pending candidates can be reviewed or edited.
3. Approving a ConceptCandidate creates an official Concept only if officialization gates pass.
4. Approving a RelationCandidate creates an official ConceptRelation only if officialization gates pass.
5. Existing official Concepts/Relations remain official only if merged evidence still passes officialization gates.
6. Candidate rejection and merge actions are recorded with review metadata.
7. Every promoted Concept or Relation keeps evidence citations.
```

## Review states

Candidate status remains:

```text
pending
approved
rejected
merged
```

`v1.5.0` adds review metadata to ConceptCandidate and RelationCandidate:

```text
reviewedBy
reviewedAt
reviewNote
promotedConceptId / promotedRelationId
mergedIntoConceptId / mergedIntoRelationId
```

## API contract

### ConceptCandidate endpoints

```http
PATCH /v1/ontology/concept-candidates/{candidateId}
POST  /v1/ontology/concept-candidates/{candidateId}/approve
POST  /v1/ontology/concept-candidates/{candidateId}/reject
POST  /v1/ontology/concept-candidates/{candidateId}/merge
```

#### Edit ConceptCandidate

Request:

```json
{
  "editedBy": "reviewer@example.com",
  "name": "Settlement Process",
  "aliases": ["settlement"],
  "proposedDefinition": "The reviewed process of finalizing obligations.",
  "conceptType": "process",
  "evidenceFragmentIds": ["evidence-id"],
  "confidence": 0.91,
  "rationale": "Reviewer clarified the candidate label."
}
```

Response:

```text
candidate
auditEventIds
```

Rules:

```text
- candidate must be pending
- editedBy must be an authorized reviewer
- replacement evidence ids must exist
- edit does not promote the candidate
```

#### Approve ConceptCandidate

Request:

```json
{
  "reviewedBy": "reviewer@example.com",
  "aliases": ["payment settlement"],
  "shortDefinition": "The process of finalizing obligations.",
  "owner": "Finance Operations",
  "reviewNote": "Definition is supported by reviewed evidence."
}
```

Response:

```text
candidate
concept
auditEventIds
```

Rules:

```text
- candidate must be pending
- reviewedBy must be authorized
- referenced EvidenceFragments must exist
- officialization gates must pass
- Concept name/alias cannot conflict with an existing Concept
- created Concept is official when approval succeeds
- candidate status becomes approved
- candidate.promotedConceptId points to the new Concept
```

#### Reject ConceptCandidate

Request:

```json
{
  "reviewedBy": "reviewer@example.com",
  "reviewNote": "Candidate does not represent a stable domain concept."
}
```

Rules:

```text
- candidate must be pending
- candidate status becomes rejected
- no Concept is created
```

#### Merge ConceptCandidate

Request:

```json
{
  "reviewedBy": "reviewer@example.com",
  "targetConceptId": "concept-id",
  "aliases": ["payment settlement"],
  "appendEvidence": true,
  "reviewNote": "Candidate is a duplicate of the existing official Concept."
}
```

Rules:

```text
- candidate must be pending
- target Concept must exist
- candidate evidence can be appended to the target Concept
- candidate name/aliases can become aliases on the target Concept
- if target is official, officialization gates must still pass after merge
- candidate status becomes merged
- candidate.mergedIntoConceptId points to the target Concept
```

### RelationCandidate endpoints

```http
PATCH /v1/ontology/relation-candidates/{candidateId}
POST  /v1/ontology/relation-candidates/{candidateId}/approve
POST  /v1/ontology/relation-candidates/{candidateId}/reject
POST  /v1/ontology/relation-candidates/{candidateId}/merge
```

#### Edit RelationCandidate

Request:

```json
{
  "editedBy": "reviewer@example.com",
  "sourceName": "Clearing",
  "targetName": "Settlement",
  "relationType": "precedes",
  "sourceConceptId": "concept-clearing",
  "targetConceptId": "concept-settlement",
  "evidenceFragmentIds": ["evidence-id"],
  "confidence": 0.88,
  "rationale": "Source explicitly states clearing precedes settlement."
}
```

Rules:

```text
- candidate must be pending
- editedBy must be authorized
- referenced evidence and Concepts must exist
- edit does not promote the candidate
```

#### Approve RelationCandidate

Request:

```json
{
  "reviewedBy": "reviewer@example.com"
}
```

Optional overrides:

```json
{
  "sourceConceptId": "concept-clearing",
  "targetConceptId": "concept-settlement",
  "relationType": "precedes"
}
```

Rules:

```text
- candidate must be pending
- source and target endpoints must resolve to existing Concepts
- source and target Concepts must be official
- evidence must be reviewed and eligible
- duplicate source/target/type Relation is blocked; use merge instead
- candidate status becomes approved
- candidate.promotedRelationId points to the new ConceptRelation
```

Endpoint resolution order:

```text
1. explicit request sourceConceptId / targetConceptId
2. candidate sourceConceptId / targetConceptId
3. promotedConceptId or mergedIntoConceptId from linked ConceptCandidates
4. exact existing Concept name or alias match
```

#### Reject RelationCandidate

Rules:

```text
- candidate must be pending
- candidate status becomes rejected
- no ConceptRelation is created
```

#### Merge RelationCandidate

Request:

```json
{
  "reviewedBy": "reviewer@example.com",
  "targetRelationId": "relation-id",
  "appendEvidence": true,
  "reviewNote": "Duplicate relation; evidence appended."
}
```

Rules:

```text
- candidate must be pending
- target ConceptRelation must exist
- candidate evidence may be appended
- if target Relation is official, officialization gates must still pass after merge
- candidate status becomes merged
- candidate.mergedIntoRelationId points to the target ConceptRelation
```

## Status codes

```text
200 OK          review action applied
403 Forbidden   reviewer is not authorized
404 Not Found   candidate, evidence, Concept, or Relation is missing
409 Conflict    candidate state, duplicate term/relation, or officialization gate blocks the action
422 Unprocessable Entity  invalid request shape
```

## Settlement proof case

Input evidence:

```text
Settlement is the process of finalizing obligations.
Clearing precedes settlement.
Reconciliation validates settlement.
```

Expected flow:

```text
1. Upload/manual-sync source text.
2. Review all generated EvidenceFragments.
3. Run ontology extraction.
4. Approve ConceptCandidate: Settlement.
5. Approve ConceptCandidate: Clearing.
6. Approve RelationCandidate: Clearing --precedes--> Settlement.
7. Query /v1/ontology/graph?concept=Settlement&depth=1&mode=official.
```

Expected graph:

```text
Settlement <--precedes-- Clearing
```

The graph edge cites the reviewed evidence fragment that supports the relation.

## Exit criteria

`v1.5.0` is ready when:

```text
1. Candidate review routes are available in the API.
2. Unauthorized reviewers receive 403.
3. Rejected candidates cannot later be approved.
4. ConceptCandidate approval creates an official Concept only through officialization gates.
5. RelationCandidate approval creates an official ConceptRelation only through officialization gates.
6. Merged candidates point to their target Concept or ConceptRelation.
7. Official graph updates only after successful review/promotion.
8. Documentation clearly states what v1.5.0 does and does not do.
```
